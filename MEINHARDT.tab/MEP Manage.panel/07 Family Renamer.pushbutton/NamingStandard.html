MHT Revit Family Naming Standard

This document outlines the standard naming convention for all Revit families used in the MHT NWBHP project. This convention is based on the principles outlined in the ISO 19650 standard for information management, adapted for Revit family naming. The goal is to create consistent, predictable, and easily searchable names that clearly identify a family's purpose at a glance.

General Naming Structure

The standard name is composed of several parts separated by hyphens. Each part provides specific information about the family.

Company MHT
-
Discipline ME
-
Category EQ
-
Description Family-Name
-
Sub-Type / Parameters Type-Name

Component Breakdown

Here is a detailed explanation of each component in the naming structure.

1. Company Identifier

The name always begins with the company initials.

  • MHT: Stands for Meinhardt.

2. Discipline Code

The second part is a two-letter code representing the primary engineering discipline for the family.

Code Discipline
ME Mechanical / HVAC
PL Plumbing / Piping
EL Electrical
FP Fire Protection
AN Annotation (Tags, Symbols, Title Blocks, etc.)

3. Category Code

The third part is a short code that identifies the Revit category of the family. This helps group similar types of components.

Code Revit Category
EQ Mechanical Equipment
PF Pipe Fittings
DF Duct Fittings
PLF Plumbing Fixtures
LFX Lighting Fixtures
EFX Electrical Fixtures
GM Generic Models
FAM Generic Annotations

4. Description (Family & Type)

This part describes the component. It is typically built from the original family name and type name, cleaned up for consistency.

  • Family Name: A descriptive name for the overall family (e.g., Air-Handling-Unit).
  • Type Name / Parameters: Describes the specific variation or size (e.g., VAV-Single-Duct, DN100, or a specific model number).

5. Smart Classification (Generic Naming)

To create cleaner and more consistent names, the tool includes a "smart classification" engine. Instead of always using the original family and type name for the description, it first attempts to identify the family's generic purpose.

  • The tool looks for clues within the family, such as its Revit Category, specific parameter names (e.g., `CFM`, `GPM`, `Head`), and keywords in the family name.
  • If a match is found based on the rules in naming_rules.json, it will use a standardized, generic name (like Pump or Air-Handling-Unit) for the descriptive part of the name.
  • If no classification rule matches, it will fall back to using the cleaned-up original family and type name as the description.

Air Terminal Naming Logic

Air Terminals have special logic to make their names more descriptive:

  1. The tool searches common type and instance parameter names (for example: System Abbreviation, Service Type, Type Comments, Comments, or other configured parameter names) for known system abbreviations such as SAD, EAD, RAD, etc. The parameter list is configurable in naming_rules.json for each classification rule.
  2. If no system type is found, the tool will insert the placeholder UND (Undefined) so names remain predictable and searchable.
  3. When multiple families would produce the same generic classified name (for example, MHT-ME-AT-SAD), the tool first attempts to create a more specific name by appending a size parameter (e.g., -DN150) when a relevant size parameter exists on the type or instance. Only if no size information is available for conflict resolution will the tool append numeric suffixes (e.g., -01, -02) as a last resort.

Hierarchical Conflict Resolution

To keep names meaningful and minimize artificial suffixes, the tool resolves conflicts in a hierarchy of specificity:

  1. Prefer different generic names when classification rules detect different sub-types.
  2. If generics collide, append system code (e.g., -SAD) when available.
  3. If still colliding, append the size parameter (e.g., -DN150) to disambiguate.
  4. Only when none of the above delivers uniqueness does the tool append incremental numeric suffixes (e.g., -01).

Exact Duplicate Family Handling

When two or more loaded families appear to be exact duplicates (same category, same number of types, and matching parameter patterns), the tool will prompt you during the Apply step with a dedicated dialog that summarizes duplicate groups and suggests actions:

  • Keep — Retain this family. The UI will smartly pre-select the family to Keep if it is already used in the project.
  • Mark as Obsolete (X_ Prefix) — Rename the family by adding an X_ prefix (for example, X_MHT-...). This is the forced action for families that are in use to avoid deleting used content.
  • Delete — Permanently delete the family from the project. This option is only enabled when the family is not used in the model.

The duplicate handler UI pre-selects sensible defaults (keep a used family, mark unused families as obsolete) but you can override choices before the apply step proceeds.

Examples in Practice

Example 1: Mechanical Equipment

An Air Handling Unit family named "AHU - Rooftop" with a type named "1200 CFM".

Resulting Name: MHT-ME-EQ-AHU-Rooftop-1200-CFM

  • MHT: Company
  • ME: Mechanical discipline
  • EQ: Mechanical Equipment category
  • AHU-Rooftop-1200-CFM: Descriptive name from the family and type.

Example 1.1: Smart Classification

A pump family with a long, vendor-specific name like "Pump-Inline-Circulator-Grundfos-TP000-SEEK".

Resulting Name: MHT-ME-MEQ-PUMP

  • MHT: Company
  • ME: Mechanical discipline
  • MEQ: Mechanical Equipment category code from the template.
  • PUMP: The generic name identified by the smart classification engine. The original family/type name is replaced for consistency.

Example 2: Pipe Fitting

A pipe elbow family with a type that has a "Size" parameter of "100mm".

Resulting Name: MHT-PL-PF-Elbow-Generic-DN100

  • MHT: Company
  • PL: Plumbing discipline
  • PF: Pipe Fitting category
  • Elbow-Generic-DN100: Description including the nominal diameter (DN) from the size parameter.

Example 3: Annotation Tag

A tag for Mechanical Equipment.

Resulting Name: MHT-AN-TAG-Mechanical-Equipment

  • MHT: Company
  • AN: Annotation discipline
  • TAG: A common identifier for tags.
  • Mechanical-Equipment: Describes what the tag is for.
